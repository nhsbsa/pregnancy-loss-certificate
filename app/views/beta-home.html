Version 15 (Multiple loss twins and triplets)

Main parent/applicant start
Other parent start
What we created
We created a journey that reflects the twin and triplet baby loss, by allowing these users to request two or more certificates in one application without reaching the end of the journey
  • Added content on the start page
  • Added new screen Did you have a single, twin or triplet pregnancy?
  • Added new screen Are you requesting certificates for both babies from your twin pregnancy?
  • Added new screen Are you requesting a certificate for each triplet?
  • Added application number as well as certificate reference number on the confirmation page Request pending
What we tested
  • The content on the start page
  • The content on the new twin and triplet loss screens
  • The user journey
What we found
  • Users had a good understanding of the application number and certificate reference numbers.
  • The question ‘Have you lost both babies from your twin pregnancy?’ was clear for most users.
Any known issues?
Overall most users were happy with the journey, however most people we tested with would assume that both or more certificates would come in the same envelope, this is something we will keep an eye on from the survey to see if this would be a need. One user out of all that we tested still assumed the twin or triplet details would be on the same certificate, again the content states otherwise but this is something we plan on monitoring. One user struggled with the question "have you lost both babies from your twin pregnancy?" as the options did not reflect their experience.

Version 14 (Still send certififcate preference and historic loss removal)

Main parent/applicant start
Other parent start
What we created
We created a journey that allows the main applicant to still receive a certificate in their name only or to be cancelled if the other parent cannot confirm or doesnt want to be on the certificate. We also removed the question
  • Added new screen Would you still like to receive the certificate if the other parent does not respond?
  • Removal of screen Did your baby loss happen on or after 1 September 2018?
What we tested
  • The still send certificate option if the other parent doesn't reply, where in the journey would this screen fit best?
What we found
  • Users understood the purpose and wording of the “still send” question. When selecting ‘yes’, all expected to receive a certificate even if the other parent did not respond, and two expected an email confirmation.
  • For the ‘no’ option, three participants believed the request would be cancelled if the other parent didn’t respond. Two others expected to receive a certificate with the other parent field removed, though they hadn’t selected ‘no’ and didn’t see the cancellation message.
  • Most participants felt the question appeared at the right point in the user journey, though one preferred it earlier.
Any known issues?
Overall most of the users understood the question and liked the placement of the question within the journey. However one user raised concerns that it was unclear what would appear on the certificate some expected it to be blank and others would assume it would be removed. With this in mind we added body text underneath the yes and no radios making them conditional to give clear instructions on what each option meant.

Version 12 (Recurrent and multiple loss) and sample certificate

Parent 1 Start screen
Other parent start screen
What we created
We created a journey to allow parents with recurrent losses (meaning baby loss over a period of time) to be able to request for multiple certificates without going through identity verficiation again as well as not entering the second parents details again if they want to add the same parent. They will not need to go through identity verification if they choose to add a second parent or proceed on a single application journey.
  • Added new screen Would you like to request a certificate for another baby loss?
  • Added new screen Would you like to add this other parent to the next certificate?
  • Added new screen Would you like to add the same other parent to the next certificate?
  • Added "continue" button on the request pending screen Request pending
What we tested
  • the flow to allow users to apply for another certificate without going through identity verification again
  • if the new flow made sense to users
  • if the content on these screens was clear and made sense
  • if users understand they can continue onto the next certificate after the request pending screen
  • We tested two versions of the request pending screen one with the continue button against the request pending screen with the question "would you like to request a certificate for another baby loss?"
What we found
  • Users understood the overall content of the screens and questions, however the flow didn't make sense to some of the users and were confused as to why they had to answer some eligibility questions again.
Any known issues?
The request pending screen felt like an end point for users and they felt like they had already answered the question that they would like another certificate so why would they continue from that screen. Some users would miss the continue button on the first journey and just exit the journey. Users preferred the option of having the request pending screen with the question as one screen.

Scenario 2

Main applicant does their own ID verification and NOT do second parents ID Verification. Second parent completes a prompted journey where they verify who they are, ultimately giving consent. This is an asynchronous journey! (2nd parent gets a notification via email or SMS that contains a link to start the journey.)
